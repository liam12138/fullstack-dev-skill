## 简介

全栈开发工程师 Skill 是一款智能化的项目开发引导工具，旨在帮助开发者从零开始系统化地完成项目初始化、需求分析、技术方案设计和代码实现。类似于 OpenSpec 的理念，但更加聚焦于实际代码生成环节。

本 Skill 兼容 Trae IDE 和 Claude Code 两个主流开发工具，能够自动识别当前平台并适配相应的交互方式。无论您使用哪个平台，都能获得一致的使用体验。

### 核心价值

- **系统化开发流程**：提供从需求到代码的完整开发路径，避免遗漏关键步骤
- **智能化方案推荐**：基于项目需求自动推荐最合适的技术方案
- **对话式交互**：通过自然对话引导开发决策，支持随时调整和回退
- **多架构支持**：支持单体应用、前后端分离、微服务等多种架构模式
- **跨平台兼容**：无缝集成 Trae 和 Claude Code

## 功能特性

### 1. 项目初始化引导

系统提供交互式的项目初始化流程，通过结构化问答收集项目基本信息：

- 项目名称和描述
- 项目类型（Web 应用、移动应用、后端服务等）
- 架构模式选择（单体、前后端分离、微服务）
- 团队规模和技术背景

### 2. 需求收集与分析

通过多轮对话系统性地收集项目需求：

- **功能需求**：核心业务功能、用户角色、数据管理等
- **非功能需求**：性能指标、安全要求、兼容性需求
- **约束条件**：预算限制、时间安排、技术约束

### 3. 技术方案生成

智能分析需求并生成完整的技术方案文档：

- 技术选型建议与对比分析
- 系统架构设计与模块划分
- API 接口设计规范
- 数据库设计方案
- 部署架构规划

### 4. 多应用架构支持

对于前后端分离和微服务架构，提供更强大的支持：

- **前后端分离架构**
  - 前端应用独立技术方案
  - 后端服务独立技术方案
  - API 契约文档自动生成

- **微服务架构**
  - 各微服务独立技术文档
  - 服务治理方案（注册发现、配置中心、链路追踪）
  - 服务间通信协议设计
  - API 网关方案

### 5. 代码自动生成

根据技术方案自动生成项目基础代码结构：

- 多种技术栈模板支持（React、Vue、Node.js、Python、Java 等）
- 标准化的目录结构
- 基础组件和配置文件的自动生成
- README 和项目文档生成

### 6. 对话式方案确认

在关键决策点与用户进行对话确认：

- 多轮确认流程（架构类型 → 技术栈 → 关键设计 → 整体方案）
- 方案解释和对比分析
- 修改意见即时反馈
- 完整的状态回退支持

## 快速开始

### 方式一：使用命令触发

在终端或命令面板中输入以下命令之一：

```
/init
/new-project
/fullstack
```

或者使用中文触发短语：

- "帮我初始化一个新项目"
- "创建一个新项目"
- "开始一个新项目"
- "帮我设计技术方案"
- "生成项目代码"

### 方式二：逐步引导

1. **启动 Skill**：输入 `/init` 或描述您的需求
2. **回答问题**：按照引导完成项目基本信息填写
3. **确认初始化**：项目配置确认后，进入需求收集阶段
4. **收集需求**：详细描述您的功能和非功能需求
5. **生成文档**：系统自动生成需求文档和技术方案
6. **方案确认**：审查并确认技术方案
7. **生成代码**：系统根据方案生成初始代码结构

## 使用指南

### 项目初始化

启动项目初始化流程，系统会依次询问以下信息：

```
Q1: 项目名称
   请输入项目名称，将用于生成项目目录和配置文件
   
Q2: 项目描述
   简要描述项目的用途和目标
   
Q3: 项目类型
   - Web 应用
   - 移动应用
   - 后端服务
   - 桌面应用
   - 命令行工具
   - 类库/SDK
   
Q4: 架构类型
   （仅在选择 Web 或后端服务时显示）
   - 单体应用
   - 前后端分离
   - 微服务架构
   
Q5: 团队规模
   - 个人开发
   - 小团队（2-5人）
   - 中型团队（6-20人）
   - 大型团队（20人以上）
   
Q6: 技术熟悉度
   选择团队熟悉的技术（可多选）
```

### 需求收集

完成初始化后，进入需求收集阶段：

#### 功能需求收集

系统会引导您详细描述：

- **目标用户**：项目面向的用户群体特征
- **核心功能**：系统需要实现的主要功能模块
- **用户角色**：系统中的用户类型及其权限
- **数据管理**：数据存储和管理需求

#### 非功能需求收集

收集性能、安全等非功能性要求：

- **预期用户量**：日活、月活用户数量
- **性能指标**：响应时间、吞吐量要求
- **安全要求**：认证、授权、加密需求
- **兼容性**：浏览器、设备兼容要求

#### 约束条件

- **预算限制**：低/中/高预算
- **开发周期**：项目交付时间要求
- **技术约束**：必须使用或不能使用的技术

### 技术方案对话

在技术方案阶段，系统会：

1. **推荐架构类型**
   - 展示推荐的架构模式
   - 解释推荐理由
   - 提供备选方案对比

2. **确认技术栈**
   - 前端框架选择
   - 后端技术选型
   - 数据库方案
   - 缓存策略

3. **讨论关键设计**
   - API 设计风格
   - 认证授权方案
   - 数据存储策略

您可以随时提问或要求修改，系统会根据反馈调整方案。

### 多应用架构处理

对于前后端分离和微服务架构，系统会：

#### 前后端分离架构

1. **分别收集前后端需求**
   - 前端：UI 框架偏好、状态管理需求、路由要求
   - 后端：API 风格、数据访问模式、业务逻辑复杂度

2. **生成独立技术方案**
   - 前端应用技术方案文档
   - 后端服务技术方案文档

3. **生成 API 契约文档**
   - API 接口规范
   - 数据交换格式
   - 错误处理约定

#### 微服务架构

1. **划分服务边界**
   - 根据业务领域划分服务
   - 确定服务间依赖关系
   - 设计服务通信协议

2. **生成服务技术文档**
   - 每个微服务独立的技术方案
   - 服务间接口定义

3. **生成整体架构文档**
   - 服务拓扑图
   - 服务治理方案
   - 部署架构

### 代码生成

确认技术方案后，系统会：

1. **选择代码模板**
   根据技术方案选择对应的模板

2. **生成项目结构**
   创建目录结构和基础文件

3. **生成核心代码**
   - 配置文件
   - 数据模型
   - API 接口框架
   - 基础组件

4. **生成文档**
   - README.md
   - 项目说明文档
   - 开发指南

## 工作流程

### 标准工作流程

```
┌─────────────┐
│  项目初始化  │  →  收集项目基本信息
└──────┬──────┘
       ↓
┌─────────────┐
│  需求收集   │  →  功能和非功能需求
└──────┬──────┘
       ↓
┌─────────────┐
│  技术方案   │  →  架构设计和选型
└──────┬──────┘
       ↓
┌─────────────┐
│  代码生成   │  →  生成项目代码
└──────┬──────┘
       ↓
┌─────────────┐
│  项目交付   │  →  完成初始化
└─────────────┘
```

### 多应用架构工作流程

```
┌─────────────┐
│  架构选择   │  →  确定架构类型
└──────┬──────┘
       ↓
┌──────────────────────────────────┐
│        应用/服务边界划分          │
└──────┬─────────────────────┬────┘
       ↓                     ↓
┌─────────────┐      ┌─────────────┐
│  应用 A     │      │  应用 B     │
│  需求收集   │      │  需求收集   │
└──────┬──────┘      └──────┬──────┘
       ↓                     ↓
┌─────────────┐      ┌─────────────┐
│  技术方案   │      │  技术方案   │
└──────┬──────┘      └──────┬──────┘
       ↓                     ↓
┌──────────────────────────────────┐
│         整体架构文档              │
└──────────────────────────────────┘
```

### 回退机制

支持在任意阶段回退到之前的决策点：

```
当前阶段: 技术方案
      ↓
可用回退点:
  - 需求收集（修改需求）
  - 项目初始化（修改基本信息）
```

## 配置选项

### 全局配置

在 `skill.json` 中配置 Skill 全局选项：

```json
{
  "workflow": {
    "stages": ["init", "requirements", "tech-spec", "code-gen", "delivery"],
    "confirmationRequired": true,
    "allowRollback": true
  }
}
```

### 技术栈默认值

```json
{
  "defaults": {
    "frontend": {
      "framework": "react",
      "language": "typescript",
      "buildTool": "vite"
    },
    "backend": {
      "framework": "nodejs",
      "language": "typescript"
    }
  }
}
```

### 文档输出配置

```json
{
  "output": {
    "format": "markdown",
    "includeDiagrams": true,
    "language": "zh-CN"
  }
}
```

## 常见问题

### Q1: 如何修改已确认的方案？

在任意阶段输入 `修改`，系统会提供修改选项。您可以：

- 选择需要修改的部分
- 进行针对性修改
- 重新确认修改后的方案

### Q2: 支持哪些技术栈？

**前端**

- React + TypeScript + Vite
- Vue 3 + TypeScript + Vite
- Angular + TypeScript
- Next.js / Nuxt.js (SSR)

**后端**

- Node.js + Express/NestJS
- Python + FastAPI/Django
- Java + Spring Boot
- Go + Gin

**数据库**

- PostgreSQL, MySQL, MongoDB, SQLite

**缓存**

- Redis, Memcached

### Q3: 如何处理微服务架构？

1. 在架构选择阶段选择"微服务架构"
2. 系统会引导您划分服务边界
3. 为每个服务单独收集需求和生成技术方案
4. 系统生成整体架构文档和服务治理方案

### Q4: 代码生成后如何继续开发？

生成的代码是完整的初始项目结构，包含：

- 基础配置和依赖
- 数据模型定义
- API 接口框架
- 基础组件
- README 文档

您可以直接在生成的项目基础上继续开发，或根据需要调整代码结构。

### Q5: 如何在不同平台间切换？

Skill 会自动检测当前平台并适配交互方式：

- **Trae IDE**：利用 Skill 机制，支持自然语言交互
- **Claude Code**：通过 MCP 协议，支持命令行交互

两个平台功能一致，您可以根据习惯选择使用。

### Q6: 回退后数据会丢失吗？

不会。系统会保存每个阶段的完整数据，回退只是跳转到之前的阶段，您之前填写的信息不会丢失。回退后可以直接修改需要调整的部分，无需重新填写所有信息。

### Q7: 如何添加自定义代码模板？

1. 在 `templates` 目录下创建新的模板文件夹
2. 定义模板的 `template.json` 配置文件
3. 创建模板文件（使用模板变量）
4. 在 `templates.json` 中注册新模板

### Q8: 生成的文档可以自定义格式吗？

可以。文档生成器使用模板系统，您可以：

- 修改 `src/doc-generator/templates` 中的模板文件
- 自定义 Markdown 格式和章节结构
- 添加或删除特定章节
